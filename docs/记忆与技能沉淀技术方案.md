# Open Jarvis 记忆与技能沉淀技术方案

> 记忆系统让 AI 跨会话保持经验，技能沉淀让高频经验自动升级为全局可复用的能力

**相关文档**：[AGENTS.md §12.13–12.15](../AGENTS.md) · [源码导览 §5.4 记忆面板](源码导览.md) · 代码版本 **0.1.0**

---

## 一、设计目标

| 目标 | 说明 |
|------|------|
| 经验持久化 | 对话结束后可将关键经验沉淀为 Markdown 文件（**默认关闭**，需在设置中开启） |
| 智能召回 | Agent 在执行任务前自动检索并读取相关记忆，避免重复踩坑 |
| 自驱晋升 | 高频记忆达到阈值后自动提示用户沉淀为全局技能，无需手动配置 |
| 可逆可控 | 沉淀、晋升、撤销全链路可逆，用户始终拥有最终决策权 |
| 工作区隔离 | 记忆按工作区存储，技能全局共享 |

## 三、数据模型

### 3.1 记忆文件格式

记忆以 Markdown + YAML Frontmatter 形式存储在工作区目录下。

**存储路径**: `<workspace>/.open-jarvis/memories/<topic>.md`

**路由前缀**: `/memories/`（Agent 通过虚拟路径访问）

```markdown
---
title: "前端项目部署流程"
summary: "使用CI/CD将前端项目部署到生产环境的标准化流程"
keywords:
  - deploy
  - ci-cd
  - frontend
workspaceTags:
  - /Users/name/my-project
recallCount: 5
lastRecalledAt: "2026-05-27T10:00:00.000Z"
lastUpdatedAt: "2026-05-27T10:00:00.000Z"
promotedSkillFolder: null
promotionStatus: none
---

# 前端项目部署流程

## 适用场景
当任务涉及 deploy、ci-cd、frontend 时优先使用这个技能。

## 操作流程
1. 检查 package.json 中的 build 脚本
2. 确认环境变量配置
3. 执行构建命令
4. ...

## 注意事项
- 部署前必须确认 .env.production 存在
- 构建产物目录默认为 dist/
...
```

### 3.2 记忆 Frontmatter 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `title` | string | 通用化主题标题 |
| `summary` | string | 一句话摘要 |
| `keywords` | string[] | 3-8 个关键词，用于相似度计算 |
| `workspaceTags` | string[] | 工作区路径标签 |
| `recallCount` | number | 被召回次数，驱动晋升流程 |
| `lastRecalledAt` | string\|null | 最后召回时间（ISO 格式） |
| `lastUpdatedAt` | string | 最后更新时间 |
| `promotedSkillFolder` | string\|null | 晋升后的技能文件夹名 |
| `promotionStatus` | enum | 晋升状态：`none` → `candidate` → `promoted` / `rejected` |

### 3.3 技能文件格式

技能以独立文件夹形式存储在全局目录。

**存储路径**: `~/.open-jarvis/skills/<skill-name>/SKILL.md`

```markdown
---
name: "deploy-workflow"
description: "前端项目部署流程 的通用处理方法、操作流程与注意事项。"
---

# 前端项目部署流程

本技能应以通用、可复用的方式理解和使用，避免把它当成一次性任务记录。

## 适用场景
当任务涉及 deploy、ci-cd、frontend 或相近问题模式时优先使用这个技能。

## 来源记忆
- 记忆路径: /memories/deploy-workflow.md
- 召回次数: 5

## 操作流程
从下面的工作方法中提炼稳定、可迁移的执行顺序、关键检查点和必要命令模式。

## 注意事项
从下面的工作方法中提炼通用的风险点、边界条件和回退办法。

## 工作方法
（记忆正文原样搬入）
```

### 3.4 类型定义

```typescript

// 技能使用快照
interface SkillUsageSnapshot {
  items: SkillUsageItem[];

  occurredAt: string;
}

## 四、记忆召回（Recall）

### 4.1 召回流程

```
┌─────────────────────────────────────────────────────────────┐
│                    MemoryRecallMiddleware                     │
│                                                             │
│  触发时机：Agent 每次调用 LLM 前                              │
│                                                             │
│  ┌──────────────────────┐                                   │
│  │ 检查已有 ToolMessage │                                   │
│  │ 是否包含 /memories/  │──▶ 是 ──▶ 跳过，不重复注入         │
│  └──────────┬───────────┘                                   │
│             │ 否                                             │
│             ▼                                                │
│  ┌──────────────────────────────────────────────────┐       │
│  │ 追加记忆提醒到系统提示词：                          │       │
│  │                                                   │       │
│  │ "Before you plan, edit files, or run              │       │
│  │  substantive commands, inspect the persistent     │       │
│  │  memory directory."                               │       │
│  │                                                   │       │
│  │ 1. Run ls("/memories/") to see available memories │       │
│  │ 2. Read any relevant memory files under           │       │
│  │    /memories/ before continuing                   │       │
│  │ 3. If no relevant memory exists, continue         │       │
│  │    normally after that quick check                │       │
│  └──────────────────────────────────────────────────┘       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│                   Agent 自主行为                              │
│                                                             │
│  ls("/memories/")  ──▶  read_file("/memories/deploy.md")   │
│                                                             │
│  （Agent 使用内置文件系统工具主动检索和阅读记忆）              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│                  召回追踪（对话结束后）                        │
│                                                             │
│  遍历对话历史中所有 read_file 工具调用                        │
│  提取路径包含 /memories/ 的调用                              │
│  对被召回的记忆文件：                                        │
│    - recallCount += 1                                       │
│    - lastRecalledAt = 当前时间                               │
│    - 构建 MemoryRecallSnapshot                              │
│    - 推送 memory_recall 事件到前端                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 关键设计决策

- **被动引导而非主动注入**：Middleware 不直接把记忆内容塞进上下文，而是提醒 Agent 先 `ls` 再 `read_file`。这避免了把大量不相关记忆灌入上下文窗口，让 Agent 自主判断哪些记忆与当前任务相关
- **去重保护**：如果对话历史中已存在包含 `/memories/` 路径的 ToolMessage，则不再注入提醒，防止循环提示
- **召回计数**：对话结束后统计本轮被实际读取的记忆文件，递增 `recallCount`，这是晋升流程的输入

## 五、记忆沉淀（Consolidation）

### 5.1 触发时机

```
用户发送消息                    用户审批/恢复执行
     │                               │
     ▼                               ▼
agent:invoke ──▶ Agent 流式执行 ──▶ agent:resume ──▶ Agent 继续执行
                                         │
                                         │ 流结束且未被中断
                                         ▼
                            ┌─────────────────────────┐
                            │  后台模型异步执行沉淀     │
                            │  consolidateTaskMemory() │
                            │  （不阻塞用户界面）       │
                            └─────────────────────────┘
```

沉淀在 Agent 流完成后由后台模型异步执行，不阻塞用户交互。使用 `getBackgroundTaskModel(modelId)` 获取独立于对话主模型的后台 LLM 实例。

### 5.2 沉淀完整流程

```
┌──────────────────────────────────────────────────────────────────┐
│                   consolidateTaskMemory()                        │
│                                                                  │
│  ① extractConversationTranscript(state)                         │
│     ├── 从 graph state 提取完整消息序列                           │
│     └── 序列化为纯文本对话记录                                    │
│                                                                  │
│  ② extractRecalledMemoryPaths(state)                            │
│     ├── 遍历所有消息的 tool_calls                                 │
│     ├── 筛选 read_file 调用中包含 /memories/ 的路径               │
│     └── 去重后返回被召回的记忆路由路径列表                         │
│                                                                  │
│  ③ updateRecallCounts(workspacePath, recalledPaths)             │
│     ├── 对每个被召回路径，recallCount += 1                       │
│     └── 更新 lastRecalledAt 为当前时间                           │
│                                                                  │
│  ④ buildMemoryRecallSnapshot(workspacePath, recalledPaths)      │
│     └── 构建 MemoryRecallSnapshot（含 items + totalCount + 时间）│
│                                                                  │
│  ⑤ listMemoryDocuments(workspacePath)                           │
│     ├── 遍历 .open-jarvis/memories/ 下所有 .md 文件              │
│     └── 解析每个文件的 frontmatter，返回 MemoryDocumentRecord[]   │
│                                                                  │
│  ⑥ generateMemoryDraft(model, transcript, existingDocs, lang)  │
│     ├── 使用 MEMORY_SYSTEM_PROMPT 指导 LLM                      │
│     ├── LLM 输出严格 JSON：title/summary/keywords/target_path/  │
│     │                            body_markdown                  │
│     └── parseGeneratedDraft(text) 解析 JSON                      │
│                                                                  │
│  ⑦ resolveMemoryRoutePath(draft, existingDocs, recalledPaths)   │
│     ├── 优先级1: 本轮被召回的现有记忆 → 更新它                   │
│     ├── 优先级2: 精确路径匹配 → 更新它                           │
│     ├── 优先级3: token Jaccard 相似度 >= 0.35 → 合并到它         │
│     └── 优先级4: 创建新文件                                      │
│                                                                  │
│  ⑧ writeMemoryDocument(workspacePath, routePath, frontmatter,  │
│                          bodyMarkdown)                           │
│     ├── 确保 .open-jarvis/memories/ 目录存在                     │
│     ├── 序列化 frontmatter + body 为完整 Markdown                │
│     └── 写入文件（创建或覆盖）                                    │
│                                                                  │
│  ⑨ maybeBuildPromotionCandidate(workspacePath, routePath)      │
│     ├── 读取写入后的 frontmatter                                 │
│     ├── 检查 recallCount >= threshold                            │
│     ├── 排除已 promoted / rejected                                │
│     ├── 若达标：标记 promotionStatus = "candidate"               │
│     └── 返回 MemoryPromotionCandidate 或 null                   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.3 LLM 沉淀提示词（MEMORY_SYSTEM_PROMPT）

核心规则：

```
You consolidate completed agent work into durable topic memory.

Return strict JSON only with these keys:
- title: short generalized topic title
- summary: one-sentence generalized summary
- keywords: array of 3-8 short keywords
- target_path: an absolute route path under /memories/ ending in .md
- body_markdown: markdown body only, without frontmatter

Rules:
- Prefer editing an existing similar memory instead of creating duplicates
- Write in generalized reusable language (not one-off task reports)
- Avoid exact user requests, timestamps, single-run outcomes, absolute paths
- Prefer abstract topics: capability, workflow, pattern, failure mode
- Focus on scenarios, successful workflow, concrete execution steps,
  failure signals, commands/files/constraints
- The markdown body MUST explicitly preserve the task's operation
  flow/process and cautions/notes so it can be reused as a skill later
- Default output language: Chinese (unless English content dominates)
```

### 5.4 智能路径匹配（resolveMemoryRoutePath）

```

                          │
                          ▼
              ┌───────────────────────┐
              │ 本轮被召回的记忆路径？ │

### 5.5 相似度算法

采用基于 token 重叠的 Jaccard 相似度，无需向量数据库：

```
┌─────────────────────────────────────────────────────────────┐
│                    相似度计算流程                              │
│                                                             │
│  draft:   title + summary + keywords                        │
│  document: title + summary + keywords                      │
│                                                             │
│  ① tokenizeForSimilarity(text)                             │
│     ├── 按非字母数字中文字符分割                               │
│     ├── 过滤长度 < 2 的 token                               │
│     └── 返回 token 数组                                     │
│                                                             │
│  ② 构建 Set                                                │
│     draftTokens = new Set([...titleToks, ...summaryToks,   │
│                           ...keywordToks])                  │
│     docTokens   = new Set([...titleToks, ...summaryToks,   │
│                           ...keywordToks])                  │
│                                                             │
│  ③ Jaccard 系数                                             │
│     overlap = draftTokens ∩ docTokens                       │
│     score   = |overlap| / max(|draft|, |doc|)              │
│                                                             │
│  ④ 判定                                                    │
│     score >= 0.35 → 合并到现有记忆                           │
│     score <  0.35 → 创建新文件                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

特点：
- 中英文混合分词：`[一-鿿]` 中文字符单独处理
- 无外部依赖：不需要嵌入模型或向量数据库
- 0.35 阈值：经过实践调优，在"合并相似"和"保持区分"之间取得平衡

## 六、技能晋升（Promotion）

### 6.1 晋升状态机

```
                    用户每次对话
                        │
                        ▼
              ┌─────────────────┐
              │  recallCount++  │
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────────┐
              │ recallCount < 阈值?  │
              └──────┬──────┬───────┘
                     │      │
                   是      否
                     │      │
                     ▼      ▼
              继续积累   ┌─────────────────────┐
                        │ status == promoted?  │
                        │ 或 rejected?        │
                        └──────┬──────┬───────┘
                               │      │
                             是      否
                               │      │
                               ▼      ▼
                          不再触发  promotionStatus = "candidate"
                                        │
                                        ▼
                                 推送晋升候选事件到前端
                                        │
                              ┌─────────┼─────────┐
                              ▼         ▼         ▼
                          确认晋升   暂不沉淀    拒绝晋升
                              │         │         │
                              ▼         │         ▼
                    promotionStatus    │  promotionStatus
                      = "promoted"    │    = "rejected"
                              │       │         │
                              ▼       │    不再提示
                    创建技能文件      │
                    (~/.open-jarvis/  │
                     skills/<slug>/  │
                     SKILL.md)       │
                              │       │
                              ▼       │
                    promotedSkillFolder│
                      = "<slug>"     │
                                     │
                                     ▼
                                  下次对话
                                  recallCount 继续累积
                                  但不再触发晋升提示
```

### 6.2 记忆设置

| 参数 | 默认值 | 范围 | 配置位置 | 说明 |
|------|--------|------|---------|------|
| `memoryConsolidationEnabled` | **false** | boolean | electron-store `settings.memorySettings` | 关闭时不调用 `consolidateTaskMemory` |
| `skillPromotionRecallThreshold` | 3 | 1-50 | electron-store `settings.memorySettings` | 召回次数达到阈值后提示晋升 |

**UI 入口**：设置 → 记忆（`MemoryPanel.tsx`）中的「启用记忆沉淀」开关；晋升阈值同面板可调。

阈值含义：记忆被召回 3 次后，系统认为该经验具有高频复用价值，建议沉淀为全局技能。

## 十四、设计权衡

| 决策 | 选择 | 备选 | 理由 |
|------|------|------|------|
| 记忆存储 | Markdown 文件 + frontmatter | SQLite / 向量数据库 | 可读性、可编辑性、可 git 追踪；小规模场景无需向量索引 |
| 召回方式 | Middleware 提醒 + Agent 自主 read_file | 语义检索 + 自动注入上下文 | 避免 Token 浪费，让 Agent 自主判断相关性 |
| 相似度算法 | token Jaccard 系数 | 余弦相似度 / 嵌入向量 | 无外部依赖，零配置，0.35 阈值已调优 |
| 沉淀时机 | 对话结束后后台异步（opt-in） | 对话中实时 | 默认关闭，用户显式开启；不阻塞交互 |
| 晋升触发 | recallCount 累积阈值 | 时间衰减 / 人工标记 | 简单可量化，用户可配置阈值 |
| 技能目录 | 全局共享 | 工作区级隔离 | 技能本质是跨项目可复用的能力，全局更合理 |
| 语言策略 | 中英文检测 + 默认中文 | 强制中文 / 强制英文 | 适配用户实际使用场景 |
