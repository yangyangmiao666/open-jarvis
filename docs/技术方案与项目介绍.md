# Open Jarvis 技术方案与项目介绍

> An open-source alternative to Claude Cowork | Apache-2.0 License

## 一、项目定位

Open Jarvis 是一款强大的跨平台桌面智能体工作台。它让用户通过自然语言对话，把任务直接落到绑定工作区和本地环境中执行，例如读写文件、运行代码、整理资料、调用 MCP 工具和操作浏览器。核心设计原则：**AI 不只聊天，而是动手做事，并且所有动作都可见、可审查、可回放**。

项目当前聚焦三个目标：

- **工作区协作**：每个会话绑定独立工作区，AI 围绕当前目录进行分析、修改与执行
- **本地执行**：文件访问、命令运行、技能加载和大多数状态都在本机完成，不依赖远端代理
- **可控自动化**：通过工具调用、审批中断、执行日志和持久化状态，让自动化过程始终处于用户控制之下

```
┌─────────────────────────────────────────────┐
│                  用户                        │
│                "帮我整理下载目录"              │
└──────────────────┬──────────────────────────┘
                   │ 自然语言
                   ▼
┌─────────────────────────────────────────────┐
│              Open Jarvis                     │
│                                             │
│   理解意图 → 制定计划 → 调用工具 → 执行动作    │
│                                             │
│   整理下载目录：                              │
│   1. 扫描下载文件夹                           │
│   2. 按类型分类                               │
│   3. 移动文件到对应子目录                       │
└──────────────────┬──────────────────────────┘
                   │ 本地操作
                   ▼
┌─────────────────────────────────────────────┐
│              用户电脑                        │
│         文件系统 / 终端 / 浏览器               │
└─────────────────────────────────────────────┘
```

用户说一句话，AI 理解后拆解为可执行步骤，每一步通过工具在本地真实执行，结果实时反馈给用户。AI 还能跨会话积累经验，高频经验自动晋升为全局技能。

## 二、整体架构

应用采用 Electron 桌面架构，分为三个核心层：

```
┌──────────────────────────────────────────────────────────┐
│                       界面层（渲染进程）                    │
│                                                          │
│   对话界面  │  工作区文件 │  模型与技能 │  审批与设置         │
└────────────────────────┬─────────────────────────────────┘
                         │ 双向 IPC 通信
                         ▼
┌──────────────────────────────────────────────────────────┐
│                       核心层（主进程）                     │
│                                                          │
│   对话引擎  │  工具调度  │  技能系统  │  本地沙箱管理       │
│   MCP 桥接  │  代理配置  │  权限审查  │  会话管理           │
└────────────────────────┬─────────────────────────────────┘
                         │ 调用
                         ▼
┌──────────────────────────────────────────────────────────┐
│                       执行层                              │
│                                                          │
│   本地文件系统  │  工作区本地沙箱 │  浏览器  │  外部 API      │
└──────────────────────────────────────────────────────────┘
```

- **界面层**：Electron 渲染进程，React + TypeScript 构建，负责所有用户交互
- **核心层**：Electron 主进程，承载对话引擎、工具调度、技能加载、MCP 桥接、权限审查等核心逻辑
- **执行层**：实际执行操作的环境，包括本地文件系统、工作区本地沙箱、浏览器以及模型与 MCP 所依赖的外部接口

三层之间通过 Electron 的 IPC 机制通信，界面层不直接操作文件系统或执行命令，所有操作由核心层统一调度。

## 三、对话引擎

对话引擎是整个应用的大脑，负责与 AI 模型交互、解析工具调用、管理对话上下文。

### 3.1 工作流程

```
用户输入
  │
  ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 构造提示词 │───▶│ 调用模型  │───▶│ 解析响应  │
└──────────┘    └──────────┘    └────┬─────┘
                                     │
                          ┌──────────┴──────────┐
                          ▼                     ▼
                     纯文本回复             工具调用请求
                     直接展示              交给工具调度器
                                              │
                                              ▼
                                        ┌──────────┐
                                        │ 执行工具  │
                                        └────┬─────┘
                                             │
                                             ▼
                                        将结果追加到对话
                                             │
                                             ▼
                                        再次调用模型 ◀──┐
                                             │            │
                                    ┌────────┴───┐        │
                                    ▼            ▼        │
                               文本回复     还要调工具 ─────┘
                               展示给用户
```

一次用户输入可能触发多轮模型调用。模型在每轮中可以请求调用工具，工具执行结果反馈给模型后，模型决定是继续调用还是给出最终回复。这种"思考→行动→观察"的循环就是 Agent 的核心模式。

### 3.2 上下文管理

对话历史会随轮次增长，但模型上下文窗口有限。引擎采用窗口裁剪策略：

- 保留系统提示词和技能描述（始终在上下文中）
- 保留最近 N 轮对话
- 超出部分从最早的消息开始裁剪
- 工具调用的输入输出体积较大时，单独做截断处理

### 3.3 多模型支持

引擎通过统一的适配层对接不同 AI 服务商：

```
┌───────────────────────────────────┐
│            对话引擎                │
│         （统一调用接口）            │
└─────────────┬─────────────────────┘
              │
     ┌────────┼────────┬──────────┐
     ▼        ▼        ▼          ▼
  OpenAI   Anthropic  Google    本地模型
   GPT      Claude    Gemini    Ollama
```

每个模型服务商实现相同的接口，引擎不关心底层差异。用户可在设置中切换默认模型，也可在对话中临时指定。

## 四、模型适配层

不同 AI 服务商的 API 格式存在差异，适配层负责抹平这些差异，让上层引擎用统一的方式调用任何模型。

### 4.1 两种核心格式

当前业界主要有两种模型 API 格式：

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   OpenAI 格式                              Anthropic 格式     │
│   ──────────                               ──────────        │
│                                                              │
│   工具定义放在 tools 数组中                   工具定义放在       │
│   每个工具是 function 类型                    tools 数组中      │
│   调用结果用 role=tool 的消息返回             调用结果用         │
│   流式响应用 delta.content                   role=user 的      │
│   逐块拼接                                    content 块返回   │
│                                              流式响应用         │
│                                              content_block     │
│                                              逐块拼接          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

关键差异集中在三个方面：

| 维度 | OpenAI 格式 | Anthropic 格式 |
|------|-----------|---------------|
| 工具调用结果 | 独立的 `tool` 角色消息 | 嵌入 `user` 角色的 `tool_result` 块 |
| 流式响应结构 | `delta.content` 文本片段 | `content_block_delta` 事件流 |
| 思考/推理 | `reasoning_content` 字段 | `thinking` 内容块类型 |

### 4.2 适配器架构

```
┌────────────────────────────────────────────┐
│              对话引擎                       │
│         （只认识统一格式）                    │
└──────────────────┬─────────────────────────┘
                   │
                   ▼
┌────────────────────────────────────────────┐
│              适配器接口                     │
│                                            │
│   统一方法：                                │
│   - 发送消息（返回流式响应）                  │
│   - 解析工具调用                            │
│   - 构造工具结果消息                         │
│   - 解析思考内容                            │
└──────────────────┬─────────────────────────┘
                   │
      ┌────────────┼────────────┐
      ▼            ▼            ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ OpenAI   │ │Anthropic │ │  Google  │
│ 适配器   │ │  适配器   │ │  适配器   │
└──────────┘ └──────────┘ └──────────┘
      │            │            │
      ▼            ▼            ▼
   GPT 系列    Claude 系列   Gemini 系列
   + 兼容接口   + 原生接口    + 原生接口
   (Ollama等)  (思考模式)
```

每个适配器负责：
- **请求转换**：将统一格式的对话历史和工具定义，转换为目标 API 的请求格式
- **响应解析**：将目标 API 的流式响应，解析为统一的文本片段、工具调用、思考内容三种事件
- **消息构造**：将工具执行结果构造成目标 API 能理解的格式，追加到对话历史中

### 4.3 OpenAI 兼容生态

OpenAI 格式已成为事实标准，大量模型服务（Ollama、vLLM、DeepSeek、通义千问等）都提供 OpenAI 兼容接口。适配器只需配置不同的 `baseURL`，即可接入这些服务：

```
┌──────────────────────────────────────────┐
│           OpenAI 兼容适配器               │
│                                          │
│   baseURL: https://api.openai.com/v1     │  → GPT 系列
│   baseURL: http://localhost:11434/v1     │  → Ollama 本地模型
│   baseURL: https://api.deepseek.com/v1   │  → DeepSeek
│   baseURL: https://dashscope.aliyuncs... │  → 通义千问
│   ...                                    │
└──────────────────────────────────────────┘
```

用户在设置中填入 API 地址和密钥，适配器自动识别并连接。

### 4.4 思考深度配置

部分模型（如 Claude、DeepSeek）支持"深度思考"模式，让模型在回答前先进行内部推理。思考深度影响模型花多少算力来推理：

```
┌──────────────────────────────────────────────────────┐
│                思考深度等级                             │
│                                                      │
│   低            中            高            最高        │
│   ◄──────────────────────────────────────────────▶   │
│                                                      │
│   快速回答      适度推理      深度分析      极致推理     │
│   适合简单问题   适合常规任务   适合复杂逻辑   适合难题    │
│   响应快        均衡           响应慢        最慢       │
│   消耗少        消耗适中       消耗高        消耗最高    │
│                                                      │
└──────────────────────────────────────────────────────┘
```

实现方式因模型而异：
- **Anthropic**：通过 `thinking` 参数和 `budget_tokens`（思考预算）控制，预算越高思考越深入
- **OpenAI**：通过 `reasoning_effort` 参数（low / medium / high）控制
- **其他模型**：部分模型不支持此功能，适配器会忽略该参数

思考内容对用户可见：在界面中以折叠区域展示模型的推理过程，帮助用户理解 AI 的决策逻辑。

## 五、工具系统

工具是 AI 与本地环境交互的唯一通道。模型不能直接操作文件或执行命令，只能通过调用工具间接完成。

### 5.1 工具分类

```
┌─────────────────────────────────────────────────────┐
│                     工具体系                          │
├──────────────┬──────────────┬───────────────────────┤
│   文件操作    │   命令执行    │      信息获取          │
├──────────────┼──────────────┼───────────────────────┤
│  读取文件     │ 执行终端命令  │  网页搜索              │
│  写入文件     │ 运行代码     │  网页内容提取           │
│  列出目录     │              │  系统信息查询           │
│  移动/删除    │              │                       │
└──────────────┴──────────────┴───────────────────────┘
```

- **文件操作**：读写文件、列出目录内容、移动和删除文件
- **命令执行**：在沙箱或本地终端中执行命令，运行代码脚本
- **信息获取**：搜索互联网、提取网页内容、查询系统状态

### 5.2 工具调用流程

```
模型输出："我要读取 /Users/xxx/notes.txt"
       │
       ▼
┌──────────────┐
│  解析工具调用  │   从模型响应中提取工具名和参数
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  权限检查     │   该工具是否需要用户确认？
└──────┬───────┘
       │
       ├── 需要确认 ──▶ 弹窗询问用户 ──▶ 允许/拒绝
       │                                    │
       ▼                                    ▼
┌──────────────┐                    ┌──────────────┐
│  执行工具     │                    │  返回拒绝信息  │
└──────┬───────┘                    └──────────────┘
       │
       ▼
┌──────────────┐
│  返回执行结果  │   结果追加到对话，交给模型继续处理
└──────────────┘
```

## 六、人工审查（HITL）

AI 自主执行操作时，某些动作可能带来不可逆的后果。人工审查（Human-in-the-Loop）机制让用户在关键节点介入，决定操作是否继续。

### 6.1 审查触发条件

```
┌─────────────────────────────────────────────────────────┐
│                    操作风险分级                           │
│                                                         │
│  低风险                    中风险               高风险    │
│  ◄───────────────────────────────────────────────────▶  │
│                                                         │
│  读取文件    写入文件    执行命令    删除文件    网络请求   │
│  列出目录    创建文件    安装依赖    修改配置    访问外部   │
│                                                         │
│  ◄─── 自动执行 ───▶│◄──── 需确认 ──────────────────▶   │
│                    │                                    │
│                用户可调整分界线                            │
└─────────────────────────────────────────────────────────┘
```

### 6.2 审查交互流程

```
AI 请求执行高风险操作
       │
       ▼
┌──────────────────────────────────────────────┐
│              审查弹窗                         │
│                                              │
│  操作：删除 /Users/xxx/old_project 目录       │
│  理由：该目录已确认不再需要                     │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  允许一次  │  │ 本次拒绝  │  │ 始终允许  │   │
│  └──────────┘  └──────────┘  └──────────┘   │
└──────────────────────────────────────────────┘
       │
       ├── 允许一次 → 执行本次操作，下次仍需确认
       ├── 本次拒绝 → 告知 AI 操作被拒绝，AI 调整方案
       └── 始终允许 → 该类操作以后自动执行
```

三种审查级别：

| 级别 | 行为 | 适用场景 |
|------|------|---------|
| 全部确认 | 每个工具调用都需用户批准 | 首次使用、敏感环境 |
| 仅高风险 | 读取类自动执行，写入/执行类需确认 | 日常使用（默认） |
| 全部自动 | 所有操作自动执行，不弹窗 | 信任环境、批量任务 |

审查决策通过 IPC 从渲染进程传回主进程，主进程根据结果决定是否继续执行工具。

## 七、技能系统

技能是对工具的封装和编排。单个工具能力有限（如"读文件"），技能将多个工具组合成有意义的业务流程（如"代码审查"）。

### 7.1 技能的结构

```
┌─────────────────────────────────────┐
│              一个技能                 │
├─────────────────────────────────────┤
│                                     │
│  名称：代码审查                      │
│  描述：审查指定文件或目录的代码质量    │
│                                     │
│  ┌─────────────────────────────┐    │
│  │  提示词模板                  │    │
│  │                             │    │
│  │  "你是一个代码审查专家，      │    │
│  │   请审查以下代码：            │    │
│  │   {{文件内容}}"              │    │
│  └─────────────────────────────┘    │
│                                     │
│  ┌─────────────────────────────┐    │
│  │  可用工具列表                 │    │
│  │                             │    │
│  │  读取文件、列出目录、         │    │
│  │  执行命令                    │    │
│  └─────────────────────────────┘    │
│                                     │
└─────────────────────────────────────┘
```

每个技能包含：
- **提示词模板**：定义 AI 在该技能下的角色和行为规则，支持变量插值
- **工具列表**：该技能允许 AI 使用的工具子集，限制能力范围
- **元信息**：名称、描述、图标等，用于界面展示

### 7.2 技能加载流程

```
┌──────────────────────┐
│   全局技能目录         │
│   ~/.open-jarvis/     │
│      skills/          │
│        代码审查/       │
│        文档生成/       │
│        数据分析/       │
└──────────┬───────────┘
           │ 启动时扫描
           ▼
┌──────────────────────┐
│   技能注册表          │
│                      │
│   验证格式合法性      │
│   加载提示词模板      │
│   注册可用工具        │
└──────────┬───────────┘
           │ 注入对话
           ▼
┌──────────────────────┐
│   对话上下文          │
│                      │
│   系统提示词中包含     │
│   所有已激活技能的     │
│   名称和描述          │
└──────────────────────┘
```

技能存放在全局目录 `~/.open-jarvis/skills/` 下，应用启动时自动扫描并加载。用户可在界面中启用或禁用技能，只有启用的技能才会注入对话上下文。

### 7.3 技能与对话的关系

```
用户选择技能："代码审查"
       │
       ▼
┌──────────────────────────────────────┐
│  构造对话上下文                        │
│                                      │
│  系统提示词 = 默认角色 + 技能提示词     │
│  可用工具   = 默认工具 ∩ 技能工具       │
└──────────────────────────────────────┘
       │
       ▼
  对话开始，AI 以该技能的角色行事
```

选择技能后，技能的提示词会叠加到系统提示词中，AI 的行为被限定在该技能定义的角色和规则内。

## 八、记忆与技能沉淀系统

AI 助手最大的问题之一是"每次对话都从零开始"。用户反复遇到相同的坑、反复教 AI 相同的经验，效率极低。记忆系统解决了这个问题：每次对话的关键经验自动沉淀为持久化文件，下次对话时 AI 自主检索并复用；高频记忆达到阈值后自动晋升为全局技能，跨项目、跨工作区可用。

### 8.1 记忆生命周期

```
┌──────────────────────────────────────────────────────────┐
│                     记忆完整生命周期                        │
│                                                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  │  召回     │──▶│  沉淀     │──▶│  晋升候选 │──▶│  技能沉淀 │
│  │          │   │          │   │          │   │          │
│  │ Agent 自 │   │ 对话结束 │   │ recallCo │   │ 用户确认 │
│  │ 主检索并 │   │ 后台模型 │   │ unt >= 3 │   │ 或手动沉 │
│  │ 读取相关 │   │ 自动生成 │   │ 时标记   │   │ 淀为全局 │
│  │ 记忆文件 │   │ 记忆草稿 │   │ candidate│   │ 技能     │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘
│                                                          │
│                              ┌──────────┐   ┌──────────┐
│                              │  撤销沉淀 │◀──│  拒绝晋升 │
│                              │          │   │          │
│                              │ 删除技能 │   │ 标记     │
│                              │ 文件+重  │   │ rejected │
│                              │ 置状态   │   │ 不再提示 │
│                              └──────────┘   └──────────┘
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 8.2 记忆召回机制

AI 在执行任务前，不是被动等待记忆被塞进上下文，而是被引导主动检索：

```
┌──────────────────────────────────────────────────────┐
│              MemoryRecallMiddleware                    │
│                                                      │
│  每次模型调用前，检查对话历史中是否已有记忆相关内容   │
│                                                      │
│  若没有，追加提醒到系统提示词：                        │
│                                                      │
│  "Before you plan, edit files, or run                │
│   substantive commands, inspect the persistent       │
│   memory directory."                                 │
│                                                      │
│  1. Run ls("/memories/")                             │
│  2. Read relevant files before continuing             │
│  3. If no relevant memory, continue normally         │
│                                                      │
└──────────────────────────────────────────────────────┘
```

Agent 随后自主执行 `ls` + `read_file` 检索记忆。对话结束后，系统统计本轮被实际读取的记忆文件，递增 `recallCount`，这是晋升流程的驱动输入。

### 8.3 记忆沉淀流程

对话结束后，后台模型异步执行沉淀（不阻塞用户交互）：

```
┌──────────────────────────────────────────────────┐
│              沉淀引擎 (后台 LLM)                  │
│                                                  │
│  ① 提取对话记录                                 │
│  ② 提取被召回的记忆路径                          │
│  ③ 更新召回计数 (recallCount++)                 │
│  ④ 列出现有记忆文档                              │
│  ⑤ LLM 生成记忆草稿 (JSON 格式)                 │
│  ⑥ 智能路径匹配：                               │
│     本轮召回 > 精确匹配 > 相似度 >= 0.35 > 新建  │
│  ⑦ 写入/更新记忆文件 (Markdown + Frontmatter)   │
│  ⑧ 检查晋升候选 (recallCount >= 阈值?)          │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 8.4 相似度匹配策略

沉淀时需要判断新草稿是合并到已有记忆还是创建新文件：

```
┌──────────────────────────────────────────────────┐
│              四级路径匹配优先级                    │
│                                                  │
│  优先级1: 本轮被召回的记忆                       │
│  └─ Agent 实际 read_file 读取过的文件优先更新   │
│                                                  │
│  优先级2: 精确路径匹配                           │
│  └─ draft.target_path 对应已有文件              │
│                                                  │
│  优先级3: token Jaccard 相似度 >= 0.35          │
│  └─ 中英文混合分词，无需向量数据库               │
│  └─ 合并到最相似的已有记忆                      │
│                                                  │
│  优先级4: 创建新记忆文件                         │
│  └─ 以上都不匹配时，创建新 Markdown 文件        │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 8.5 技能晋升与沉淀

当记忆被召回次数达到阈值（默认3次），系统自动标记为晋升候选并提示用户：

```
┌──────────────────────────────────────────────────┐
│  💡 记忆已达到技能沉淀阈值                        │
│                                                  │
│  "前端项目部署流程" 已被召回 5 次                  │
│                                                  │
│  ┌──────────┐  ┌──────────────────┐             │
│  │ 暂不沉淀  │  │ 沉淀为全局技能   │             │
│  └──────────┘  └──────────────────┘             │
│                                                  │
└──────────────────────────────────────────────────┘
```

确认后，记忆文件自动转化为全局技能文件 `~/.open-jarvis/skills/<slug>/SKILL.md`，所有工作区的 Agent 都能发现和使用。

### 8.6 记忆与技能的存储拓扑

```
<workspace>/.open-jarvis/memories/    ← 工作区级记忆
├── deploy-workflow.md
├── react-state-mgmt.md
└── api-debug-patterns.md

~/.open-jarvis/skills/                 ← 全局技能目录
├── deploy-workflow/
│   └── SKILL.md                      ← 从记忆晋升产生的技能
├── react-state-mgmt/
│   └── SKILL.md
└── custom-skill/
    └── SKILL.md                      ← 用户手动创建的技能
```

记忆按工作区隔离（每个项目有自己的经验库），技能全局共享（跨项目可复用）。详细技术方案见 [记忆与技能沉淀技术方案](记忆与技能沉淀技术方案.md)。

## 九、MCP 集成

MCP（Model Context Protocol）是一个开放协议，让 AI 应用能以标准化方式接入外部工具和数据源。Open Jarvis 作为 MCP 客户端，可以连接任意 MCP 服务器，将服务器提供的工具暴露给 AI 使用。

### 9.1 为什么需要 MCP

```
┌─────────────────────────────────────────────────────────┐
│  没有 MCP 时                                             │
│                                                         │
│  每接入一个外部服务，就要写一套适配代码                      │
│  数据库工具 → 写适配代码                                  │
│  GitHub 工具 → 写适配代码                                 │
│  Slack 工具  → 写适配代码                                 │
│  ...每个工具都要单独维护                                   │
│                                                         │
├─────────────────────────────────────────────────────────┤
│  有 MCP 后                                               │
│                                                         │
│  所有外部服务通过统一协议接入                               │
│  数据库 MCP 服务器 ──┐                                   │
│  GitHub MCP 服务器 ──┤──▶ 统一协议 ──▶ AI 直接使用        │
│  Slack MCP 服务器  ──┘                                   │
│  ...无需为每个服务写适配代码                               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

MCP 的核心价值：**一次接入，处处可用**。任何实现了 MCP 协议的服务器，都能被 Open Jarvis 直接使用。

### 8.2 MCP 架构

```
┌──────────────────────────────────────────────────────┐
│                    Open Jarvis                        │
│                                                      │
│   ┌──────────────┐      ┌──────────────────────┐     │
│   │   对话引擎    │◀────▶│    MCP 客户端管理器    │     │
│   └──────────────┘      └──────────┬───────────┘     │
│                                    │                  │
└────────────────────────────────────┼──────────────────┘
                                     │ 标准协议通信
                    ┌────────────────┼────────────────┐
                    ▼                ▼                ▼
             ┌──────────┐    ┌──────────┐    ┌──────────┐
             │ MCP 服务器│    │ MCP 服务器│    │ MCP 服务器│
             │  数据库   │    │  GitHub  │    │  文件系统 │
             └──────────┘    └──────────┘    └──────────┘
```

### 8.3 MCP 工具的发现与使用

```
┌────────────────┐
│  MCP 服务器启动  │
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  工具发现       │   客户端请求服务器列出所有可用工具
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  工具注册       │   将 MCP 工具转换为 AI 可调用的工具格式
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  注入对话       │   MCP 工具与内置工具一起提供给 AI
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  AI 调用 MCP 工具│  AI 选择调用某个 MCP 工具
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  转发到 MCP 服务器│  客户端将调用请求转发给对应服务器
└───────┬────────┘
        │
        ▼
┌────────────────┐
│  返回执行结果    │  服务器执行后返回结果，追加到对话
└────────────────┘
```

MCP 工具与内置工具在 AI 看来没有区别——都是可调用的工具。区别仅在于执行路径：内置工具由主进程直接执行，MCP 工具由主进程转发给 MCP 服务器执行。

### 9.3 MCP 服务器配置

用户在设置中配置 MCP 服务器连接信息：

```
┌──────────────────────────────────────────────────┐
│  MCP 服务器配置                                    │
│                                                  │
│  服务器名称：github-tools                         │
│  传输方式：标准输入输出（本地进程）或 SSE（远程服务）│
│  启动命令：npx @modelcontextprotocol/server-github │
│  环境变量：GITHUB_TOKEN=xxx                       │
│                                                  │
└──────────────────────────────────────────────────┘
```

两种传输方式：
- **标准输入输出**：MCP 服务器作为本地子进程启动，通过 stdin/stdout 通信，适合本地工具
- **SSE（Server-Sent Events）**：连接远程 MCP 服务器，通过 HTTP 通信，适合云端服务

## 十一、工作区本地沙箱

Open Jarvis 的“沙箱”不是 Docker 或容器，而是主进程基于当前工作区构造的本地执行环境。命令仍然在用户机器上以子进程方式运行，但执行目录固定在工作区，文件访问通过统一文件系统后端完成，并注入应用自带的 bun、uv 和 Python 3.12.13。再叠加审批、超时、输出截断和路径约束，形成一个更可控的本地工作区执行模型。

### 10.1 为什么需要工作区沙箱

```
┌──────────────────────────────────────────────────┐
│  没有工作区沙箱的风险                              │
│                                                  │
│  AI 生成命令 ──▶ 直接使用系统环境执行               │
│                     │                            │
│                     ├── 工作目录不可控             │
│                     ├── 依赖版本因机器而异         │
│                     ├── 长命令可能长时间挂起       │
│                     └── 高风险操作缺少统一审批     │
│                                                  │
├──────────────────────────────────────────────────┤
│  工作区沙箱提供的保护                              │
│                                                  │
│  AI 生成命令 ──▶ 进入 LocalSandbox                │
│                     │                            │
│                     ├── 工作目录锁定为当前工作区   │
│                     ├── bun/uv/Python 由应用内置   │
│                     ├── execute 工具默认走审批     │
│                     └── 120s 超时 + 100KB 输出截断 │
└──────────────────────────────────────────────────┘
```

这里要强调一点：它提升的是**可控性、一致性和可审计性**，不是操作系统级的强隔离边界。因此高风险命令仍然依赖 HITL 审批和工作区权限约束，而不是依赖容器逃逸防护这类假设。

### 10.2 沙箱架构

```
┌──────────────────────────────────────────────────────┐
│                     主进程                            │
│                                                      │
│   ┌──────────────┐     ┌──────────────────────────┐  │
│   │ Agent Runtime │────▶│      LocalSandbox        │  │
│   └──────────────┘     └──────────┬───────────────┘  │
│                                   │                  │
└───────────────────────────────────┼──────────────────┘
                                    │
              ┌─────────────────────┼─────────────────────┐
              │                     │                     │
              ▼                     ▼                     ▼
┌────────────────────┐  ┌────────────────────┐  ┌──────────────────┐
│ 当前工作区目录       │  │ 嵌入式运行时        │  │ 本地子进程执行层    │
│ rootDir = workspace│  │ bun / uv / Python │  │ shell / exec      │
│ .venv / node_modules│ │ OPEN_JARVIS_* env  │  │ stdout / stderr   │
└────────────────────┘  └────────────────────┘  └──────────────────┘
```

这个架构的关键点：

- **工作区绑定**：每个线程创建运行时时都会绑定一个绝对工作区路径，LocalSandbox 在该目录下执行命令
- **运行时注入**：应用启动后从 resources/tooling 或开发态目录解析 bun、uv 和 Python 路径，避免依赖系统安装
- **平台适配**：macOS/Linux 通过 shell wrapper 注入运行时，Windows 通过 `.open-jarvis/runtime-bin` 和 `.open-jarvis/python-runtime-bin` shim 重写命令入口
- **统一文件后端**：文件读写、glob、grep、readRaw 与 execute 共享同一后端，文本解码统一走 UTF-8 优先、GB18030 回退

### 10.3 代码执行流程

```
AI 请求执行命令："用 Python 分析这个 CSV"
       │
       ▼
┌──────────────┐
│  审批 execute │   命令执行默认进入 HITL 审批
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  规范化入口   │   阻止直接 python/node/npm 调用
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  准备运行时   │   注入 bun/uv/Python，必要时创建 .venv
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  本地子进程执行│   cwd 固定为工作区，捕获 stdout/stderr
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  返回执行结果 │   输出 + 退出码 + 超时/截断信息
└──────┬───────┘
       │
       ▼
  结果追加到对话，AI 再决定下一步
```

运行时策略是显式约束的：

- **Python**：必须通过内置 `uv` 和工作区 `.venv` 运行，禁止直接调用 `python`、`pip`、`pytest`
- **JavaScript/TypeScript**：必须通过内置 `bun` 执行，禁止直接调用 `node`、`npm`、`npx`、`pnpm`、`yarn` 等入口
- **输出治理**：默认超时 120 秒，输出上限约 100KB，防止长时间阻塞和超大日志淹没上下文

### 10.4 沙箱生命周期

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│  创建实例  │──▶│  准备运行时 │──▶│  执行/复用  │──▶│  结束清理  │
│          │   │          │   │          │   │          │
│绑定工作区  │   │解析内置工具链│   │多次执行命令  │   │释放子进程  │
│生成 sandbox│   │创建 .venv  │   │持续返回结果  │   │关闭相关资源│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
```

LocalSandbox 负责：

- **绑定目录**：将文件读写和命令执行都限制在当前工作区语义下
- **准备运行时**：解析并注入 `OPEN_JARVIS_UV`、`OPEN_JARVIS_BUN`、`OPEN_JARVIS_PYTHON` 等环境变量
- **维护 Python 环境**：在工作区内部创建或复用 `.venv`，把项目依赖留在工作区，而不是写入应用包
- **执行治理**：统一处理超时、中断、输出截断、代理变量大小写别名和日志记录

与容器模型不同，这个生命周期不会“销毁沙箱文件系统”，因为工作本身就是围绕用户真实工作区进行的。清理的重点是子进程、临时 shim 和运行时状态，而不是删除用户目录。

## 十二、DeepAgents 引擎

DeepAgents 是 Open Jarvis 的核心执行引擎，基于 LangChain 的 DeepAgents 框架构建。它将"对话式 AI"升级为"可编程的自主代理"，让 AI 能够规划任务、调用工具、管理状态，并在复杂任务中自主推进。

### 11.1 什么是 DeepAgents

传统对话 AI 是"一问一答"模式：用户问，AI 答。DeepAgents 引入了"代理循环"——AI 可以自主决定下一步做什么，直到任务完成。

```
传统对话：
  用户 ──▶ AI 回答 ──▶ 结束

DeepAgents 代理循环：
  用户 ──▶ AI 思考 ──▶ 调工具 ──▶ 观察结果 ──▶ 再思考 ──▶ ... ──▶ 最终回答
```

### 11.2 Harness（线束）架构

Harness 是 DeepAgents 的核心概念。它像一个"线束"一样，把 AI 模型、工具、权限、记忆等能力编织在一起，形成一个完整的代理。

```
┌──────────────────────────────────────────────────────────┐
│                     Harness 线束                          │
│                                                          │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│   │  AI 模型  │  │  工具集   │  │  权限控制  │              │
│   └──────────┘  └──────────┘  └──────────┘              │
│                                                          │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│   │  记忆系统  │  │  技能注入  │  │  规划能力  │              │
│   └──────────┘  └──────────┘  └──────────┘              │
│                                                          │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│   │  文件系统  │  │  沙箱执行  │  │  人工审查  │              │
│   └──────────┘  └──────────┘  └──────────┘              │
│                                                          │
│   所有能力通过 Harness 统一编排，AI 通过统一接口访问        │
└──────────────────────────────────────────────────────────┘
```

Harness 的设计理念：**能力即配置**。不需要为每种能力写不同的代码，只需在 Harness 中声明"这个代理具备哪些能力"，框架自动完成编排。

### 11.3 Harness 能力详解

```
┌─────────────────────────────────────────────────────────────┐
│                    Harness 能力矩阵                           │
├─────────────┬───────────────────────────────────────────────┤
│   规划能力   │ AI 在执行复杂任务前，先制定步骤计划              │
│             │ 计划可随执行结果动态调整                         │
├─────────────┼───────────────────────────────────────────────┤
│   工具调用   │ AI 选择合适的工具执行操作                       │
│             │ 工具结果自动反馈给 AI                           │
├─────────────┼───────────────────────────────────────────────┤
│   文件系统   │ AI 可以读写用户文件系统                        │
│             │ 操作受权限系统管控                              │
├─────────────┼───────────────────────────────────────────────┤
│   沙箱执行   │ AI 生成的命令在工作区本地沙箱中执行             │
│             │ 通过内置 bun / uv / Python 统一运行时            │
├─────────────┼───────────────────────────────────────────────┤
│   人工审查   │ 高风险操作暂停执行，等待用户确认                 │
│             │ 用户可批准、拒绝或修改操作                       │
├─────────────┼───────────────────────────────────────────────┤
│   技能注入   │ 将技能的提示词和工具集注入对话上下文             │
│             │ AI 的行为被限定在技能定义的范围内                │
├─────────────┼───────────────────────────────────────────────┤
│   记忆系统   │ 代理可以记住之前的对话和操作                    │
│             │ 跨会话保持上下文                                │
└─────────────┴───────────────────────────────────────────────┘
```

### 11.4 代理执行循环

DeepAgents 的执行是一个"规划-执行-观察"的迭代循环：

```
┌──────────────────────────────────────────────────────┐
│                  代理执行循环                           │
│                                                      │
│   ┌─────────┐                                        │
│   │  规划    │  分析任务，制定执行计划                   │
│   │         │  "我需要先读取文件，再分析内容"            │
│   └────┬────┘                                        │
│        │                                             │
│        ▼                                             │
│   ┌─────────┐                                        │
│   │  执行    │  调用工具执行计划中的步骤                  │
│   │         │  "读取文件 /Users/xxx/data.csv"           │
│   └────┬────┘                                        │
│        │                                             │
│        ▼                                             │
│   ┌─────────┐                                        │
│   │  观察    │  获取工具执行结果，更新认知                │
│   │         │  "文件包含 1000 行销售数据"                │
│   └────┬────┘                                        │
│        │                                             │
│        ▼                                             │
│   ┌─────────┐                                        │
│   │  反思    │  评估进展，决定是否调整计划                │
│   │         │  "数据格式与预期不同，需要先清洗"           │
│   └────┬────┘                                        │
│        │                                             │
│        ▼                                             │
│   任务完成？                                          │
│     ├── 是 ──▶ 输出最终结果                            │
│     └── 否 ──▶ 回到"规划" ──────────────────────┘     │
│                                                      │
└──────────────────────────────────────────────────────┘
```

与简单的"思考-行动"循环相比，DeepAgents 增加了**规划**和**反思**环节：
- **规划**：在动手之前先想清楚步骤，避免盲目执行
- **反思**：执行后评估结果，发现偏差及时调整

### 11.5 上下文工程

DeepAgents 采用"上下文工程"（Context Engineering）策略来管理对话上下文。不同于简单的"保留最近 N 条消息"，上下文工程根据任务阶段动态调整上下文内容：

```
┌────────────────────────────────────────────────────────┐
│                  上下文工程策略                          │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │  始终保留                                         │  │
│  │  - 系统提示词（代理角色定义）                       │  │
│  │  - 技能描述（当前激活的技能）                       │  │
│  │  - 工具定义（可用工具的签名和说明）                  │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │  动态调整                                         │  │
│  │  - 对话历史：保留与当前任务相关的轮次               │  │
│  │  - 工具结果：大体积输出做摘要或截断                 │  │
│  │  - 执行计划：随任务推进更新                         │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │  按需加载                                         │  │
│  │  - 文件内容：只在 AI 请求时加载                     │  │
│  │  - 历史记忆：只在相关时注入                         │  │
│  │  - MCP 工具：按服务器连接状态动态可用               │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

上下文工程的目标：在有限的上下文窗口内，让 AI 始终拥有完成当前任务所需的关键信息。

## 十三、IPC 通信机制

Electron 架构下，渲染进程（界面）和主进程（核心逻辑）运行在不同进程中，必须通过 IPC（进程间通信）交换数据。Open Jarvis 的 IPC 设计围绕"界面只展示，主进程只执行"的原则。

### 12.1 IPC 通道总览

```
┌──────────────────────────────────────────────────────────────┐
│                        IPC 通道矩阵                           │
├──────────────────┬───────────────────┬───────────────────────┤
│      分类        │      通道          │       用途            │
├──────────────────┼───────────────────┼───────────────────────┤
│   对话相关       │ 发送消息           │ 用户发送对话消息       │
│                  │ 流式响应           │ AI 回复逐字推送       │
│                  │ 中止生成           │ 用户中止当前回复       │
│                  │ 加载历史           │ 获取对话历史记录       │
├──────────────────┼───────────────────┼───────────────────────┤
│   工具相关       │ 工具调用通知       │ 主进程通知界面展示调用  │
│                  │ 工具执行结果       │ 主进程推送执行结果     │
│                  │ 权限确认请求       │ 请求用户批准高风险操作  │
│                  │ 权限确认响应       │ 用户返回批准/拒绝决定  │
├──────────────────┼───────────────────┼───────────────────────┤
│   配置相关       │ 读取配置           │ 界面请求当前配置       │
│                  │ 更新配置           │ 界面修改配置项         │
│                  │ 模型列表           │ 获取可用模型           │
├──────────────────┼───────────────────┼───────────────────────┤
│   技能相关       │ 技能列表           │ 获取已安装技能         │
│                  │ 技能开关           │ 启用/禁用技能          │
├──────────────────┼───────────────────┼───────────────────────┤
│   线程相关       │ 线程列表/历史/回滚  │ 会话列表、历史与 rewind │
├──────────────────┼───────────────────┼───────────────────────┤
│   工作区相关     │ 选择/绑定工作区     │ 维护线程 workspacePath │
│                  │ 文件列表/文本/二进制│ 加载目录与读取文件内容  │
│                  │ 文件变更通知        │ 监听 workspace:files-changed │
├──────────────────┼───────────────────┼───────────────────────┤
│   审批与设置     │ approval / settings│ 自动批准与代理配置      │
├──────────────────┼───────────────────┼───────────────────────┤
│   MCP / 技能     │ MCP 服务器列表     │ 管理扩展工具能力        │
│                  │ skills 相关接口    │ 管理技能源与工作区技能   │
└──────────────────┴───────────────────┴───────────────────────┘
```

### 12.2 流式响应的 IPC 传输

AI 回复是逐字生成的，需要流式推送到界面，让用户看到"正在打字"的效果：

```
主进程                              渲染进程
  │                                   │
  │  调用模型 API（流式）               │
  │  ◀── token 1 ────                 │
  │  ◀── token 2 ────                 │
  │  ──── token 1 ──────────────────▶ │  显示 "你"
  │  ◀── token 3 ────                 │
  │  ──── token 2 ──────────────────▶ │  显示 "你好"
  │  ◀── token 4 ────                 │
  │  ──── token 3 ──────────────────▶ │  显示 "你好，"
  │  ...                              │  ...
  │  ◀── [完成] ─────                 │
  │  ──── [完成] ───────────────────▶ │  停止打字动画
  │                                   │
```

流式传输的关键设计：
- 每个 token 到达后立即通过 IPC 推送，不做缓冲
- 推送内容区分类型：文本片段、工具调用开始、工具调用结果、思考内容、错误信息
- 用户点击"停止"时，渲染进程发送中止信号，主进程取消模型请求

### 12.3 权限确认的 IPC 交互

高风险操作需要用户确认，这是一个双向交互过程：

```
主进程                              渲染进程
  │                                   │
  │  AI 请求删除文件                   │
  │  ──── 权限确认请求 ─────────────▶ │
  │                                   │  弹出确认对话框
  │                                   │
  │  ◀──── 用户点击"允许" ─────────── │
  │                                   │
  │  执行删除操作                      │
  │  ──── 执行结果 ────────────────▶ │
  │                                   │
```

确认请求中包含操作描述、风险等级、AI 的理由，帮助用户做出知情决策。

### 12.4 Preload 安全桥接

Electron 的安全模型要求渲染进程不能直接访问 Node.js API。Preload 脚本作为安全桥梁，有选择地暴露 IPC 方法：

```
┌──────────────────────────────────────────────────────┐
│                    安全桥接                            │
│                                                      │
│  渲染进程                  主进程                     │
│  (不可访问 Node.js)        (完整 Node.js 能力)        │
│                                                      │
│      │                       │                       │
│      │   Preload 脚本        │                       │
│      │   (有限 API 暴露)     │                       │
│      │                       │                       │
│      ├── 发送消息() ─────────┤──▶ 处理并发送          │
│      ├── 接收响应() ◀────────┤──◀ 流式推送            │
│      ├── 请求确认() ◀────────┤──◀ 权限请求            │
│      ├── 确认响应() ─────────┤──▶ 用户决定            │
│      └── 读取配置() ─────────┤──▶ 返回配置            │
│                                                      │
│  渲染进程只能调用 Preload 暴露的方法，                  │
│  无法直接访问文件系统、网络等底层能力                    │
│                                                      │
└──────────────────────────────────────────────────────┘
```

Preload 通过 `contextBridge` 暴露一个受控的 API 对象，渲染进程只能调用这些预定义的方法，无法绕过安全边界。

## 十四、多会话并行处理

用户可以同时打开多个对话，每个对话独立运行，互不干扰。这使得用户可以在一个对话中执行长时间任务的同时，在另一个对话中处理其他事情。

### 13.1 并行架构

```
┌──────────────────────────────────────────────────────────────┐
│                        主进程                                 │
│                                                              │
│   ┌─────────────────┐  ┌─────────────────┐  ┌────────────┐  │
│   │    会话 A        │  │    会话 B        │  │  会话 C     │  │
│   │                 │  │                 │  │            │  │
│   │  AI 思考中...    │  │  等待用户输入    │  │  执行命令   │  │
│   │  调用工具中...   │  │                 │  │  工作区内   │  │
│   │                 │  │                 │  │            │  │
│   └────────┬────────┘  └────────┬────────┘  └─────┬──────┘  │
│            │                    │                 │         │
│            ▼                    ▼                 ▼         │
│   ┌──────────────────────────────────────────────────────┐  │
│   │              共享资源池                                │  │
│   │                                                      │  │
│   │   模型 API 配额    │  嵌入式运行时缓存 │  MCP 连接管理   │  │
│   └──────────────────────────────────────────────────────┘  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 13.2 会话隔离

每个会话拥有独立的状态，互不影响：

```
┌──────────────────────────────────────────────────────┐
│                    会话隔离策略                         │
│                                                      │
│  ┌────────────────┐  ┌────────────────┐              │
│  │   会话 A        │  │   会话 B        │              │
│  ├────────────────┤  ├────────────────┤              │
│  │ 独立对话历史    │  │ 独立对话历史    │              │
│  │ 独立工具调用    │  │ 独立工具调用    │              │
│  │ 独立代理配置    │  │ 独立代理配置    │              │
│  │ 独立工作区沙箱  │  │ 独立工作区沙箱  │              │
│  │ 独立权限状态    │  │ 独立权限状态    │              │
│  └────────────────┘  └────────────────┘              │
│                                                      │
│  共享但受控的资源：                                    │
│  - 模型 API 连接（有速率限制）                        │
│  - MCP 服务器连接（可复用）                            │
│  - 技能定义（只读，可共享）                            │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 13.3 并行执行的资源协调

```
┌────────────────────────────────────────────────────────┐
│                  资源协调器                              │
│                                                        │
│   会话 A 请求调用模型 ──▶ 检查速率限制 ──▶ 排队或执行    │
│                                                        │
│   会话 B 请求执行命令 ──▶ 检查审批与本地资源 ──▶ 排队或启动 │
│                                                        │
│   会话 C 请求 MCP 工具 ──▶ 检查连接状态 ──▶ 复用或新建   │
│                                                        │
│   原则：                                                │
│   - 不超限：遵守 API 速率限制和本地资源上限               │
│   - 不阻塞：一个会话等待时不影响其他会话                  │
│   - 可观测：用户能看到每个会话的执行状态                  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

多会话并行时，资源协调器负责：
- **API 速率限制**：多个会话共享同一个 API 密钥时，按服务商限制排队请求
- **本地执行资源预算**：多个会话并发执行命令时，共享本机 CPU、内存和 I/O，需通过超时和审批控制风险
- **MCP 连接复用**：同一 MCP 服务器只需一个连接，多个会话共享

### 14.2 配置数据

│  ┌────────────────────────────────────────────────┐  │

│  └────────────────────────────────────────────────┘  │

│  ┌────────────────────────────────────────────────┐  │
│  │  全局设置（settings.json）                       │  │
│  │  - 默认模型、OpenAI Compatible profiles          │  │

│  └────────────────────────────────────────────────┘  │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  线程元数据（openwork.sqlite）                   │  │
│  │  - thread_id、title、status                     │  │
│  │  - metadata（workspacePath 等）                 │  │
│  │  - 运行记录与助手记录                            │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  执行状态（threads/<id>.sqlite）                 │  │

│  └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 14.3 对话数据

对话数据不是简单的 JSON 文件，而是拆成“线程元数据 + 每线程 checkpoint”两层存储：

```
┌──────────────────────────────────────────────────────┐
│                  单个对话的持久化结构                    │
│                                                      │
│  openwork.sqlite：                                    │
│  - 线程基础信息、标题、状态、更新时间                  │
│  - metadata（工作区路径、模型、审批模式等）            │
│                                                      │
│  threads/<threadId>.sqlite：                          │
│  ┌────────────────────────────────────────────────┐  │
│  │  用户消息                                         │  │
│  ├────────────────────────────────────────────────┤  │
│  │  AI 回复 / AIMessageChunk                        │  │
│  ├────────────────────────────────────────────────┤  │
│  │  工具调用与工具结果                              │  │
│  ├────────────────────────────────────────────────┤  │
│  │  interrupt / todo / subagent / token usage      │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

对话数据的特点：
- **元数据与执行状态分离**：线程列表查询更快，执行状态持久化也更适合做 rewind 和恢复
- **每线程独立 checkpoint**：单个线程损坏或膨胀时不会拖垮全部会话
- **懒加载**：界面先加载线程列表，激活线程时再恢复更完整的消息与状态

### 14.4 数据流向

```
┌──────────┐     读取配置      ┌──────────────┐
│  界面     │◀───────────────│ settings.json │
│          │───────────────▶│ / .env        │
│          │     写入配置      │              │
└────┬─────┘                 └──────────────┘
       │
       │ IPC
       │
┌────▼─────┐     读写元数据     ┌──────────────┐
│  主进程   │◀───────────────│ openwork.sqlite│
│          │───────────────▶│              │
│          │     保存状态      │              │
└────┬─────┘                 └──────────────┘
       │
       │ 恢复/回滚
       │

所有用户数据仍然以 `~/.open-jarvis/` 为中心，但实现上已经演进为 SQLite + per-thread checkpoint 的组合；工作区级审批规则则放在当前项目目录的 `.open-jarvis/approval-rules.json` 中。

### 16.4 插嘴队列（Interruption Queue）

在 AI 流式响应期间，用户可以继续输入消息"插嘴"。这些消息不会立即发送给大模型，而是进入一个队列等待：

```
AI 流式响应中
       │
       ▼
用户再次输入："等等，我补充一下..."
       │
       ▼
┌──────────────────────────────────────────────┐
│              插嘴队列                         │
│                                              │
│  消息1: "补充说明"  ──  半透明 + "排队中"     │
│  消息2: "还有这个"  ──  半透明 + "排队中"     │
│                                              │
│  等待当前流式响应完成后合并发送                  │
└──────────────────────────────────────────────┘
       │
       │ AI 响应完成
       ▼
┌──────────────────────────────────────────────┐
│  队列消息合并为一条：                          │
│  "补充说明\n\n还有这个"                        │
│                                              │
│  → 作为新的用户消息发送给大模型                 │
│  → 角标消失，消息恢复正常显示                   │
└──────────────────────────────────────────────┘
```

插嘴队列的设计原则：
- **即时可见**：用户发送的消息立即出现在对话中（半透明 + "排队中"角标），不会丢失
- **自动合并**：AI 当前响应结束后，队列中的所有消息自动合并为一条发送，无需用户手动操作
- **可取消**：用户点击取消时，插嘴队列自动清空，排队消息消失
- **无缝衔接**：非 loading 状态下发送消息的行为与原来完全一致

## 十八、进程模型

### 17.1 进程结构

```
┌─────────────────────────────────────────────┐
│              Electron 主进程                  │
│                                             │
│   窗口管理  │  对话引擎  │  工具调度  │  沙箱   │
│   IPC 路由  │  会话管理  │  MCP 桥接  │  存储   │
└──────────────────────┬──────────────────────┘
                       │
            ┌──────────┼──────────┐
            ▼          ▼          ▼
┌──────────────┐ ┌──────────┐ ┌──────────┐
│  渲染进程 1   │ │ 渲染进程2 │ │  工具进程  │

```

- **主进程**：应用核心，管理窗口、处理业务逻辑、调度工具
- **渲染进程**：每个窗口一个，仅负责 UI 渲染和用户交互
- **工具进程**：沙箱代码执行在独立进程中，避免阻塞主进程

## 二十一、总结

Open Jarvis 的核心设计可以用五个关键词概括：

- **对话驱动**：用户用自然语言描述需求，AI 理解并执行
- **工具赋能**：AI 通过工具与本地环境交互，不只是聊天
- **安全隔离**：沙箱和权限系统确保 AI 的操作在可控范围内
- **经验沉淀**：记忆系统让 AI 跨会话积累经验，高频经验自动晋升为技能
- **开放集成**：MCP 协议和技能系统让 AI 能力可扩展

│            │              │            │              │            │

│  整合结果   │              │  结果反馈   │              │  环境隔离   │
└────────────┘              └────────────┘              └────────────┘
