## 1. 项目介绍

**Open Jarvis** 是一款基于 Electron 的桌面端 AI 编程助手，定位为开发者的本地智能工作空间。用户在工作区内与 AI 智能体对话，智能体可读写文件、执行命令、调用 MCP 工具、委派子代理，全程由人机协同审批（HITL）保障安全。

### 核心定位

| 维度 | 说明 |
|------|------|
| **产品形态** | Electron 桌面应用（macOS / Windows / Linux） |
| **核心价值** | 本地优先、安全可控、多模型支持、工具可扩展 |
| **目标用户** | 需要本地 AI 编程助手的开发者 |
| **品牌** | 助手名称 "Jarvis"，Gemini 风格八芒星 Logo |

### 核心能力一览

```
┌──────────────────────────────────────────────────────────────┐
│                      Open Jarvis 能力矩阵                     │
├──────────────┬───────────────────────────────────────────────┤
│  多模型对话   │ Claude / GPT / Gemini / 任意 OpenAI 兼容模型  │
│  本地沙箱     │ 文件读写 + 命令执行，嵌入式 uv/bun/Python     │

```

## 2. 项目背景

### 2.1 为什么做这个项目

当前 AI 编程助手市场存在几个核心痛点：

1. **云端依赖**：多数产品要求代码上传到云端，企业代码安全合规受限
2. **工具链割裂**：AI 助手与本地开发环境脱节，无法直接操作文件系统和运行命令
3. **模型锁定**：多数产品绑定单一模型提供商，无法按需切换
4. **审批缺失**：AI 自主执行命令缺乏安全护栏，存在误操作风险

Open Jarvis 的设计哲学是 **"本地优先、安全可控"**：

- 所有代码和命令在用户本地执行，不离开用户机器
- 工具调用默认需要人工审批，支持工作区级规则记忆
- 支持多家模型提供商，线程级切换
- 通过 MCP 协议可扩展任意工具

### 2.2 技术选型考量

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 桌面框架 | Electron 42 | 跨平台一致性、Web 技术栈复用、丰富生态 |
| 前端框架 | React 19 + Zustand | 组件化开发、轻量状态管理、社区成熟 |
| 智能体框架 | deepagents + LangGraph | LangChain 生态、检查点支持、HITL 原生集成 |
| UI 组件 | Radix UI (shadcn/ui 风格) | 无样式基础组件、可访问性、灵活定制 |
| 样式方案 | Tailwind CSS 4 | 原子化 CSS、设计系统友好、构建性能优 |
| 包管理 | Bun | 安装速度、内置运行时、与嵌入式工具链一致 |
| 检查点存储 | sql.js (纯 JS SQLite) | 无原生编译依赖、Electron 友好、每线程隔离 |
| 代码高亮 | Shiki | VS Code 级高亮质量、多语言支持、主题可配 |

### 2.3 演进历史

```
openwork (LangChain 上游)
    │
    │  fork + 重构
    ▼
open-jarvis
    ├── 品牌重塑：助手名称 → Jarvis
    ├── 嵌入式工具链：自带 uv/bun/Python，无需系统预装
    ├── Windows 运行时支持：.cmd shim 文件
    ├── 代理配置：undici 全局分发器
    ├── 主题系统：亮色/深色模式
    ├── 通知系统：Sonner toast
    ├── OpenAI 兼容增强：API 格式/思考模式/上下文窗口
    └── 安全增强：命令拦截、审批规则记忆
```

## 9. 性能设计

### 9.1 关键性能策略

| 策略 | 实现 | 效果 |
|------|------|------|
| **流式输出** | 双流模式 `["messages", "values"]`，token 级实时推送 | 用户无需等待完整响应 |
| **检查点隔离** | 每线程独立 SQLite 文件 | 线程间无 IO 竞争 |
| **检查点防膨胀** | 100MB 上限，超限自动备份重建 | 防止磁盘空间泄漏 |
| **MCP 连接缓存** | `server.id + config signature` 维度缓存 | 避免重复连接建立 |
| **文件变更防抖** | 500ms debounce | 避免高频文件事件风暴 |
| **检查点延迟写** | 100ms debounce saveToDisk | 减少 IO 次数 |
| **命令输出截断** | 100KB 上限 | 防止大输出撑爆内存 |
| **Shiki 按需加载** | 动态 import 语言包 | 避免全量加载 35+ 语言 |
| **CSV 截断预览** | 纯文本截断 | 防止大文件卡顿 |
| **窗口状态持久化** | electron-store 保存 bounds | 重启后恢复窗口位置 |

## 10. 技术亮点总结

### 10.1 嵌入式工具链：零依赖运行

终端用户无需预装 Python、Node.js、uv、bun。应用自带完整运行时，智能体可在任意工作区创建 `.venv` 并执行 Python/JS 命令。这是区别于云端 AI 编程助手的核心竞争力。

### 10.2 命令拦截与重定向：安全沙箱

直接调用 `python`/`pip`/`node`/`npm` 被自动拦截并引导到嵌入式运行时。macOS/Linux 通过 shell 函数覆盖实现，Windows 通过 `.cmd` shim 文件实现。这确保了命令执行的可控性和一致性。

### 10.3 HITL 审批记忆：渐进式信任

审批签名归一化（数字替换为 `<num>`，对象 key 排序）使得 `npm install 3.1.2` 和 `npm install 4.0.0` 产生相同签名，一次批准即可覆盖同类命令。规则持久化到工作区 `.open-jarvis/approval-rules.json`，实现跨会话的渐进式信任。

### 10.4 消息归一化：多模型兼容

OpenAI 兼容 API 的消息格式差异（array content、reasoning_content、thinking blocks、lone surrogates）通过 `OpenAICompatibleChatCompletions` 在每次请求前自动归一化，支持 DeepSeek、Qwen、GLM 等多种模型无缝接入。

### 10.5 检查点回滚：可重放的对话

每线程独立 SQLite 检查点，支持 `rewindToMessage` 截断历史并重新发送。100MB 上限自动备份重建，防止磁盘空间泄漏。

### 10.6 MCP 工具扩展：开放生态

三种传输方式（stdio / SSE / StreamableHTTP）覆盖本地进程和远程服务。连接缓存按配置签名失效，工具命名归一化 `{server}_{tool}` 避免冲突。
